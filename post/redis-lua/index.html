Redis LUA

背景

简单的需求如下

公司运营为了提高用户活跃而设计了一个积分策略，规定用户只要在指定时间段内上线打卡就有机会获得积分，积分可以换取礼物。每天初始化一定的积分，编写功能保证用户签到并且积分扣除正确。

本质

积分存储在Redis中，如何正确扣除积分问题即Redis如何正确减少问题

思路

/**
     * 判断积分>0则减1
     * @return
     */
    public Long sign(){
        if (Integer.parseInt(redisClient.get(POINT_COUNT))>0){
            return redisClient.decr(POINT_COUNT);
        }
        return -1L;
    }

存在问题

多线程同时判断积分数量即 POINT_COUNT>0 为TRUE，若此时积分只剩 1 。

那么当多线程都执行到decr操作时,积分会扣除到负数。

进一步修改 加同步锁

/**
     * 判断积分>0则减1
     * @return
     */
    public synchronized Long sign(){
        if (Integer.parseInt(redisClient.get(POINT_COUNT))>0){
            return redisClient.decr(POINT_COUNT);
        }
        return -1L;
    }

效果： 加了同步锁后，每次只有单个线程能执行sign方法，在单机环境下，可以解决并发问题。

仍然存在的问题

1.同步锁为重量级锁，类似于抢购、签到的场景下使用影响性能，对用户的体验差。

2.多机分布式的环境下，同步锁只能保证他那个服务器是线程安全的，而不能保证全局环境下的线程安全，仍然可能存在并发问题。

进一步优化

该签到步骤主要为两步：

1.从Redis中获取当前积分数POINT_COUNT

2.判断POINT_COUNT是否大于0，大于则减1，表明签到成功

问题根源

使用同步锁的目的是为了原子化两个步骤，因此除了锁之外还有其他方式吗？

LUA

Lua 脚本功能是 Reids 2.6 版本的最大亮点， 通过内嵌对 Lua 环境的支持， Redis 解决了长久以来不能高效地处理 CAS （check-and-set）命令的缺点， 并且可以通过组合使用多个命令， 轻松实现以前很难实现或者不能高效实现的模式。

/**
     * 判断积分>0则减1
     *
     * @return
     */
    public String sign() {
        String LUA = "local temp=redis.call('get','" + POINT_COUNT + "')\n" +
                "local acount=tonumber(temp)\n" +
                "if(acount>0) then\n" +
                "redis.call('decr','" + POINT_COUNT + "')\n" +
                "else\n" +
                "return -1\n" +
                "end";
        return redisClient.eval(LUA).toString();
    }

通过LUA脚本将两个步骤原子化，即使是分布式环境下，依然不会出现并发问题，而因为Redis处理网络事件是单线程的，所以每个Redis请求都是有序的处理，即可正确执行签到事

堵塞问题

LUA脚本在Redis中运行时具有原子性，一段脚本要么全部执行，要么全部不执行，并且在脚本执行的过程中，其它指令不能被插入。正是因为这个特性，需要注意，当一段脚本执行时间过程中，Redis无法对其它客户端提供服务。

简单例子

1.写个长循环脚本堵塞Redis

1) "lme"
2) "lme0225"

127.0.0.1:6379>  eval  "local a=redis.call('hget','lme0225','margin1') local b=1 repeat b=b+1 until(b>100000000000000) return a" 0

2.再打开并一个终端执行任何操作

1
2
3

127.0.0.1:6379> keys *
(error) BUSY Redis is busy running a script. You can only call SCRIPT KILL or SHUTDOWN NOSAVE.
127.0.0.1:6379>

从返回ERROR可知，执行LUA脚本会堵塞整个Redis实例中所有的缓存信息，即不能对缓存有任何操作，否则会报BUSY Redis is busy running a script. You can only call SCRIPT KILL or SHUTDOWN NOSAVE.

解决办法

修改LUA脚本执行最大时间

redis.conf中修改 lua-time-limit 的值，默认为5000（毫秒）

该配置表明执行一个脚本5秒内，其它客户端发送的指令不会被Redis接收，当超过5s，该脚本还未执行完毕，此时可以接收其它指令，当由于上个脚本未执行完成，会返回BUSY错误提示。

SCRIPT KILL or SHUTDOWN NOSAVE

KILL掉正在执行的脚本 或 执行shutdown nosave

1
2
3
4
5

127.0.0.1:6379> keys *
(error) BUSY Redis is busy running a script. You can only call SCRIPT KILL or SHUTDOWN NOSAVE.
127.0.0.1:6379> script kill
OK
127.0.0.1:6379>

重启Redis Server

特殊情况下，是无法使用SCRIPT KILL命令解决的，只能重启服务

只要脚本中涉及更新或添加操作时，由于必须保证脚本的原子性，但是更新和添加操作已经写入数据，因此脚本无法被终止,只能通过重启服务，因此在编写LUA脚本时需要格外小心。

使用LUA的好处

  1. 减少网络开销（批量执行命令）：不使用 Lua 的代码需要向 Redis 发送多次请求, 而脚本只需一次即可, 减少网络传输；
  2. 原子操作：Redis 将整个脚本作为一个原子执行，无需担心并发，也就无需事务；
  3. 复用：脚本会永久保存 Redis 中，其他客户端可继续使用。

EVAL实现

  • 1 根据客户端给定的脚本，在Lua环境中定义Lua函数
  • 2 将客户端给定的脚本保存到lua_scripts字段，进一步使用
  • 3 执行刚刚在lua中定义的函数，执行客户端给定的lua脚本

脚本执行

  • 1 将EVAL命令传入的键名参数和脚本参数保存在KEYS数组和ARGV数组中
  • 2 装载超时钩子
  • 3 执行脚本函数
  • 4 移除钩子
  • 5 结果放入缓冲区
  • 5 垃圾回收

SCRIPT LOAD 和 EVALSHA 指令

当脚本的内容很长，而且客户端需要频繁执行，那么每次都需要传递冗长的脚本内容势必比较浪费网络流量。所以 Redis 还提供了 SCRIPT LOAD 和 EVALSHA 指令来解决这个问题。

SCRIPT LOAD 指令用于将客户端提供的 lua 脚本传递到服务器而不执行，但是会得到脚本的唯一 ID，这个唯一 ID 是用来唯一标识服务器缓存的这段 lua 脚本，它是由 Redis 使用 sha1 算法揉捏脚本内容而得到的一个很长的字符串，而该字符串存储到字典中。

有了这个唯一 ID，后面客户端就可以通过 EVALSHA 指令反复执行这个脚本了。

LUA字典

lua_scripts字典

保存Lua脚本的SHA1【校验和】checksum,值是对应的脚本。

Redis会把EVAL命令执行过的脚本，或者SCRIPT LOAD加载的脚本都保存在字典中。

作用： 实现SCRIPT EXISTS命令，实现脚本的复制

脚本死循环

Redis 的指令执行是个单线程，这个单线程还要执行来自客户端的 lua 脚本。如果 lua 脚本中来一个死循环，是不是 Redis 就完蛋了？

Redis 为了解决这个问题，它提供了 script kill 指令用于动态杀死一个执行时间超时的 lua 脚本。不过 script kill 的执行有一个重要的前提，那就是当前正在执行的脚本没有对 Redis 的内部数据状态进行修改，因为 Redis 不允许 script kill 破坏脚本执行的原子性。

比如脚本内部使用了 redis.call(“set”, key, value) 修改了内部的数据，那么 script kill 执行时服务器会返回错误。下面我们来尝试以下 script kill 指令。

1
2

127.0.0.1:6379> eval 'while(true) do print("hello") end' 0

eval 指令执行后，可以明显看出来 redis 卡死了，死活没有任何响应，如果去观察 Redis 服务器日志可以看到日志在疯狂输出 hello 字符串。这时候就必须重新开启一个 redis-cli 来执行 script kill 指令。

1
2
3

127.0.0.1:6379> script kill
OK
(2.58s)

再回过头看 eval 指令的输出

1
2
3

127.0.0.1:6379> eval 'while(true) do print("hello") end' 0
(error) ERR Error running script (call to f_d395649372f578b1a0d3a1dc1b2389717cadf403): @user_script:1: Script killed by user with SCRIPT KILL...
(6.99s)

这里注意到两个疑点，第一个是 script kill 指令为什么执行了 2.58 秒，第二个是脚本都卡死了，Redis 哪里来的闲功夫接受 script kill 指令。如果你自己尝试了在第二个窗口执行 redis-cli 去连接服务器，你还会发现第三个疑点，redis-cli 建立连接有点慢，大约顿了有 1 秒左右。

Script Kill 的原理

下面我就要开始揭秘 kill 的原理了，lua 脚本引擎功能太强大了，它提供了各式各样的钩子函数，它允许在内部虚拟机执行指令时运行钩子代码。比如每执行 N 条指令执行一次某个钩子函数，Redis 正是使用了这个钩子函数。

1
2
3
4
5
6

void evalGenericCommand(client *c, int evalsha) {
  ...
  // lua引擎每执行10w条指令，执行一次钩子函数 luaMaskCountHook
  lua_sethook(lua,luaMaskCountHook,LUA_MASKCOUNT,100000);
  ...
}

Redis 在钩子函数里会忙里偷闲去处理客户端的请求，并且只有在发现 lua 脚本执行超时之后才会去处理请求，这个超时时间默认是 5 秒。于是上面提出的三个疑点也就烟消云散了。
